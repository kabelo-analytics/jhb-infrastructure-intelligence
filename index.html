01 · Case Study

What This Project Actually Supports

This project is strongest as a Python-based municipal KPI reporting example. It generates a large synthetic incident dataset, cleans linked operational tables, engineers incident-level service metrics, and surfaces summary outputs through a notebook and HTML dashboard.

Problem

Municipal incident operations are difficult to manage without a shared view of response speed, repair duration, breach rates, and backlog movement.

Focus

The current implementation supports KPI design and reporting more strongly than advanced hotspot, geospatial, or causal analysis.

Supportable Findings

The generated outputs support 24.0 hour median first response, 3.55 day median repair time, 16.4% SLA breach rate, and broad ward-level distribution rather than extreme concentration.

02 · Pipeline

Reproducible Workflow

The repo now supports a runnable workflow through direct Python execution or module execution. The implemented steps are generation, cleaning, validation, feature building, and notebook reporting.

Generate

`generator.py` creates synthetic incidents, work orders, inspections, and supporting dimension tables.

Clean

`cleaners.py` standardises selected fields, parses datetimes, and writes processed tables.

Validate

`validation.py` checks key uniqueness and foreign-key relationships across the processed model.

Engineer Features

`build_features.py` creates `first_response_hours`, `repair_days`, and `sla_breached` for incident-level analysis.

04 · Insights

Supportable Takeaways

Insight 01

Issue type should be separated in service reporting

Potholes and water leaks share the same operational model, but they do not move through it at the same repair speed. The generated data supports issue-type-specific KPI cuts more strongly than one blended repair metric.

Insight 02

Contractor benchmarking is present, but not dramatic

The current model supports contractor comparison, but the compliance range is narrow. That makes this a benchmarking view, not evidence for aggressive procurement recommendations.

Insight 03

Backlog tracking is more defensible than hotspot drama

The notebook backlog estimate remains positive week to week, which makes weekly open-case monitoring a stronger use case than unsupported hotspot escalation claims.

Insight 04

This is a solid KPI-monitoring project, not a full city-operations intelligence model

The repo demonstrates generation, cleaning, validation, feature engineering, and basic reporting well. Geospatial and repeat-incident depth are still better described as next steps than as completed analytical capability.

05 · Stack

Technical Scope

Language

Python 3.11

Core Libraries

Pandas, NumPy, Matplotlib, and notebook-based analysis. The dashboard presentation uses Chart.js.

Schema

Incidents, work orders, inspections, locations, crews, contractors, and assets are modelled as linked operational tables.

Current Limits

Hotspot logic, stronger repeat-incident logic, and geospatial outputs remain partial or planned rather than fully evidenced in the current repo.
